Two powerful modes

Search and converse — all in one widget

Bloomfire Embedded ships with two interaction modes. Both are available in the same widget — users can switch between them freely without reloading the page.

Core

Search

Semantic, intelligent search powered by Bloomfire's vector search engine. Results are ranked by relevance score — not just keyword frequency — so users always surface the most meaningful content.

  • Semantic relevance scoring on every result (e.g. 97% match)
  • Filter by content type: Posts, Series, Q&A
  • Highlighted matched terms within snippets
  • Tag-level metadata and view counts on each result
  • Related searches panel for discovery
  • One-click deep link to the full Bloomfire article
Synapse AI

Synapse

Full conversational AI with memory across turns. Users ask follow-up questions, drill into specifics, and get progressively richer answers — all in a flowing chat interface.

  • Multi-turn conversation with full context retention
  • Follow-up questions build on prior answers naturally
  • Source pills below each response for auditability
  • Typing indicator and thinking trace give transparency
  • "New conversation" button resets context cleanly
Flexible by design. Set the default mode with the mode attribute — "search" or "synapse" — or let users toggle freely within the widget. You control the experience.

Developer integration guide

Up and running in under 5 minutes

Bloomfire Embedded is a standard Web Component — no npm package, no build step, no framework dependency. If you can add a <script> tag, you can embed Bloomfire.

1

Load the widget script

Add one <script> tag to your page — typically in <head> or just before </body>. This registers the <bloomfire-search> custom element globally.

2

Place the element

Drop <bloomfire-search> anywhere in your HTML. It renders inline where you place it — in a sidebar, modal, drawer, or full-width panel. Works in React, Angular, Vue, and plain HTML.

3

Pass your credentials

Set community-id and token as HTML attributes. The widget authenticates silently — no Bloomfire login screen is ever shown to the end user.

4

Listen to events (optional)

Subscribe to the bloomfire-search custom event to capture query strings and result counts for your own analytics pipeline.

embed.html 
<!-- Step 1: Load once per page -->
<script src="https://cdn.bloomfire.com/widget/bloomfire-search.js"></script>

<!-- Step 2 + 3: Place element with credentials -->
<bloomfire-search
  mode="search"
  community-id="YOUR_COMMUNITY_ID"
  token="YOUR_AUTH_TOKEN"
></bloomfire-search>

<!-- Step 4 (optional): Analytics event listener -->
<script>
  document.querySelector('bloomfire-search')
    .addEventListener('bloomfire-search', (e) => {
      console.log('Query:', e.detail.query);
      console.log('Results:', e.detail.resultCount);
    });
</script>
Configuration reference

Attributes

Attribute Type Required Values / Default Description
community-id String Required Your community slug Identifies your Bloomfire community. All search results and AI answers are scoped to this community.
token String Required JWT or API token Auth token for the current user. Passed to the Bloomfire API silently — users never see a login screen. Generate via the Bloomfire API or SSO flow.
mode String Optional "search" · "synapse" Sets the default interaction mode. search shows semantic results with relevance scores and filters; synapse opens the full conversational AI experience. Users can switch between both modes in-widget. Defaults to "search" if omitted.
JavaScript events

Analytics integration

The widget emits a native Custom Event that you can intercept with a standard addEventListener call. Use this to forward query data to your analytics platform (Google Analytics, Segment, Mixpanel, etc.).

analytics.js 
document.querySelector('bloomfire-search')
  .addEventListener('bloomfire-search', (e) => {

    // e.detail.query       — the user's search string
    // e.detail.resultCount — number of results returned
    // e.detail.mode        — 'search' or 'synapse'

    analytics.track('Bloomfire Search', {
      query:       e.detail.query,
      results:     e.detail.resultCount,
      mode:        e.detail.mode,
      community:   e.detail.communityId,
    });
  });

Security & compliance

Enterprise-ready from day one

Bloomfire Embedded inherits the full security posture of the Bloomfire platform. No new attack surface. No new data storage. No new compliance scope.

Token-based auth

Authentication uses a short-lived JWT or API token passed as an HTML attribute. The widget never stores credentials. Token rotation and expiry are handled by your existing auth flow.

Shadow DOM isolation

The widget runs inside a Shadow DOM — its styles and scripts are fully isolated from the host page. No CSS bleed, no global namespace collisions, no XSS vectors from the host app.

Permission-scoped results

The token passed to the widget carries the user's Bloomfire permissions. Results are automatically scoped to what that user is authorized to see — community, group, and document-level permissions all apply.

No data residency impact

The widget is a UI layer only. All data remains in Bloomfire's existing storage, subject to the same retention, residency, and encryption policies already configured for your account.

Full audit trail

Every search query made through the widget is logged in Bloomfire's audit log — same as a search performed in the full product. Compliance teams have complete visibility with no gaps.

CSP compatible

The widget loads from a fixed, versioned CDN URL. Add it to your Content Security Policy whitelist once and it works. No inline scripts, no dynamic eval(), no third-party calls outside Bloomfire APIs.
